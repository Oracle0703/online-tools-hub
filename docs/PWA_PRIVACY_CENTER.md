# PWA 离线包与隐私能力中心

本文件定义 Online Tools Hub v1.0 的离线缓存、更新和隐私能力边界。实现目标不是“尽可能多地缓存”，而是让用户明确选择、可以检查、可以中止，并且让构建系统证明缓存内容只能来自当前版本的公开静态资源。

## 1. 两级离线模型

### 1.1 最小应用壳

Service Worker 安装时只预缓存首页运行闭包、离线说明页、站点清单、图标和必要的共享静态资源。它的用途是让应用能够启动、解释当前离线状态，并提供恢复路径；它不承诺所有工具和工作流已经离线可用。

最小应用壳必须满足：

- 总量不超过 2 MiB；
- 每个条目都来自当前生产构建清单；
- 条目使用构建期记录的字节数和 SHA-256 校验；
- 不因为用户浏览了一个页面就预缓存整站。

### 1.2 按需静态缓存

用户在线访问公开页面时，Service Worker 可以缓存当前版本构建清单内的同源静态页面和带内容指纹的资源。写入前必须同时满足：

- 请求方法为 `GET`；
- URL 属于当前站点 origin 和部署 base；
- URL 不含 query；
- URL 精确存在于构建期白名单；
- 响应成功，且实际字节数和 SHA-256 与白名单一致；
- 请求不是 `Range` 请求，也不携带 `Authorization`；
- 无请求正文、用户文件、Blob、运行结果或程序化数据请求。

导航请求采用 network-first；不可变的指纹资源采用 cache-first。带查询参数的导航在断网时只能进入通用离线说明页，不能去掉 query 后命中另一个缓存页面。

任何缓存命中都不是永久可信：静态资源、导航页面和离线回退页在返回前必须重新核对构建清单中的实际字节数与 SHA-256。不匹配的缓存条目立即删除，再尝试网络或安全回退。

### 1.3 完整离线包

完整离线包只能由用户主动下载。它覆盖公开工具、指南、工作流页面及其静态运行闭包，但不包含测试路由、source map、报告、Service Worker 自身或其他非公开构建文件。

构建和运行时共同执行以下硬上限：

| 边界       |   上限 |
| ---------- | -----: |
| 条目数     |    512 |
| 单项大小   | 16 MiB |
| 完整离线包 | 64 MiB |
| 最小应用壳 |  2 MiB |

下载界面展示条目数、总字节、已完成字节和进度。`navigator.storage.estimate()` 只作为容量提示；不同浏览器的估算和清理策略不同，界面不得把它描述为保证。

取消下载会停止后续网络读取，已经通过完整性验证的公开资源可以留在当前版本缓存中，重试时按缓存成员关系继续下载。用户可以显式删除完整离线包；最小应用壳仍保留，以便应用启动和说明状态。

## 2. 缓存生命周期与消息协议

最小应用壳和完整离线资源分别使用带构建版本的缓存名。Service Worker 激活时只保留当前版本缓存，旧版本不会和新白名单混用。

页面通过 `MessageChannel` 发起以下受控动作：

- 查询当前版本、总量、已完成条目和字节；
- 开始或继续下载完整离线包；
- 取消当前下载；
- 删除完整离线包。

所有消息都有协议版本和严格字段校验。Service Worker 只向发起请求的消息端口回传状态、进度、完成或稳定错误码，不把资源正文发送给页面。整个 origin 同一时刻最多运行一个下载任务；其他标签页的并发开始请求会收到稳定的 `busy` 错误码，并应稍后重试，不会合并或共享现有任务。

## 3. 版本更新

发现新 Service Worker 时，不自动刷新页面。更新面板必须说明：重新载入会丢失尚未清空的输入、输出、所选文件、批处理队列和当前进度。只有用户点击“仍要更新并重新载入”后才向等待中的 Worker 发送激活消息。

更新后完整离线包可能需要重新下载，因为新构建具有新的内容白名单和完整性摘要。界面必须把“已下载但属于旧版本”与“当前版本完整可用”区分开。

## 4. 机器可读隐私清单

生产构建发布 `/privacy-manifest.json`。清单由工具、Operation 和工作流注册表生成，不手工复制 ID；它至少声明：

- 清单格式和版本；
- 覆盖的本站代码范围，以及明确不覆盖的浏览器扩展、浏览器实现、操作系统、网络设施、托管平台日志和其他标签页或站点；
- 用户内容的本地处理、零网络发送、零持久化和零遥测；第三方依赖随构建固定打包为同源资源，运行时不从远程第三方来源动态取码；
- 允许发生的公开同源静态 `GET`；
- 剪贴板、下载和 Object URL 必须由用户主动触发的交互边界；
- 允许保存的主题、工具快捷元数据、构建白名单缓存和当前部署路径的同源 Service Worker 注册，且均不得包含用户内容；
- CSP、Worker 防护、源码扫描和注册表覆盖门禁；
- v1.0 基线十二个工具、十二个 Operation 和六个内置工作流，以及当前构建全部十三个工具、十三个 Operation 和六个内置工作流的完整清单；
- 合成隐私自检能观察和不能观察的范围。

清单使用 exact-fields 校验并稳定排序。缺少清单、字段漂移、注册表覆盖不完整、允许状态可能包含用户内容，或所需 CSP 指令缺失，都必须让生产构建失败。

## 5. 隐私能力中心与合成自检

`/privacy/` 是面向用户的隐私与能力中心，同时展示离线包入口、数据流边界、允许的浏览器状态和机器清单链接。因为 CSP 使用 `connect-src 'none'`，页面在构建时导入同一份清单数据，不在客户端重新请求它。

隐私自检只能由用户点击启动。它生成一次性随机合成数据，在当前标签页内执行代表性的真实 Operation Worker 和内置工作流，并观察：

- 当前 CSP 和 `securitypolicyviolation`；
- 运行期间加载的资源 URL；
- URL、history、Cookie、Local/Session Storage 和 IndexedDB 数据库名称；
- 当前 origin 全部 Cache Storage 名称、请求 URL，以及在单项与总量上限内读取的每个响应字节；检查不按 MIME 类型跳过二进制响应；
- Worker、Vault、Object URL 等运行资源是否释放；
- 合成值的原文、URI 编码、Base64、Base64URL 和 SHA-256 表示是否出现在上述边界。

自检不读取 IndexedDB object store 或记录值，因此机器清单和页面都把 `indexeddb-record-values` 明确列为不评估范围；它不能声称覆盖整个 origin 的全部持久化内容。自检不得返回、渲染或写入合成密文，只返回稳定结果码和检查项状态。不支持某项观察时必须显示“无法检查”，不能当作通过。IndexedDB、Cache Storage、响应 body reader 和其他宿主观察 Promise 都必须与取消/超时竞争；未完整读取的 reader 必须取消，并始终释放锁。超时、取消、`pagehide` 和异常都走统一清理路径。

界面必须原样展示以下限制：

> 自检只观察当前版本本站代码在本标签页内执行一组随机合成数据时的行为。它会读取 IndexedDB 数据库名称，但不读取其中的记录值；也不检查浏览器扩展、浏览器实现、操作系统、网络设备、托管平台日志、其他标签页或本次未执行的路径。通过不等于第三方安全认证。

## 6. 发布证据

Issue #38 的完成证据包括：

- 单元测试验证构建清单、大小上限、URL 规范化、缓存写入谓词、消息序列和失败恢复；
- 生产构建验证隐私清单、最小应用壳、完整包覆盖、字节数、摘要和 CSP；
- 生产构建扫描全部 `src` 生产源码，网络、持久化、剪贴板、PWA 与自检能力只允许命中逐文件、逐操作的精确白名单；严格 CSP 解析器同时拒绝重复指令、缺失 token 和额外 source；
- Playwright 验证下载进度、取消、继续、删除、容量提示、更新警告、断网运行、带 query 导航回退和 Cache Storage 隐私；
- 360 px、键盘、屏幕阅读器、`prefers-reduced-motion` 和 axe 门禁；
- 合成 canary 验证网络、URL、持久化状态、控制台和运行资源边界；
- Chromium、Firefox、WebKit、真实 Edge/Windows 与 Safari/macOS 候选发布门禁。
